# Реестр приказов и распоряжений на ответственных лиц

Отдельный лист (вкладка «Приказы и распоряжения») для ведения реестра
документов, назначающих ответственных лиц. Записи автоматически подтягиваются
в исполнительную документацию (ИД) по **дате окончания работ**.

## Модель данных (`Order`)

| Поле | Описание |
|------|----------|
| `kind` | тип: `приказ` / `распоряжение` |
| `number` | номер документа (обязательно) |
| `date` | дата издания, ISO (обязательно) |
| `organization` | организация, издавшая документ |
| `role` | роль подписанта (см. ниже, обязательно) |
| `fio` | ФИО ответственного лица (обязательно) |
| `position` | должность |
| `validFrom` | действует с (необяз.; пусто = без нижней границы) |
| `validTo` | действует по (необяз.; пусто = бессрочно) |

Хранение — `backend/data/orders.json` (gitignored, создаётся при запуске).

## Роли и привязка к закладкам

`backend/src/config/orders.ts` → `ORDER_ROLES`. Каждая роль маппится на пару
закладок строительных актов `Должность_<role>` / `ФИО_<role>`:

`ТНЗ, Г, ТНГ, Пр, Пд, И1, И2, И3`

## Логика подбора по дате (`resolveByDate`)

Для каждой роли среди её приказов:
1. **Приоритет** — приказы, чей интервал `[validFrom, validTo]` покрывает дату
   окончания работ; берётся с самым поздним `validFrom` (последнее назначение).
2. **Fallback** — если интервал не покрыл дату, берётся ближайший приказ,
   изданный не позже даты (помечается `fallback: true`).

Результат подстановки в закладку должности:
`<должность>, <приказ|распоряжение> № <номер> от <дата>`.

## API

| Метод | Маршрут | Назначение |
|-------|---------|------------|
| `GET` | `/api/orders/roles` | список ролей |
| `GET` | `/api/orders` | весь реестр |
| `POST` | `/api/orders` | добавить запись |
| `PUT` | `/api/orders/:id` | изменить запись |
| `DELETE` | `/api/orders/:id` | удалить запись |
| `POST` | `/api/orders/resolve` | `{ date, roles? }` → подобранные лица + поля закладок |

## Frontend

- Вкладка **«Приказы и распоряжения»** — CRUD-таблица реестра.
- В форме документа (если есть поле «Дата окончания работ») — кнопка
  **«Подтянуть ответственных»**: вызывает `resolve` и автозаполняет поля
  подписантов, показывая, какие приказы подобраны и предупреждения по fallback.
