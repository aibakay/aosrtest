# ROADMAP — объект-центричная пакетная генерация (реестры ИД)

Дорожная карта развития веб-генератора по предложению заказчика.
Дата: 2026-06-29.

## Контекст

Текущее приложение **stateless**: выбрал тип документа → заполнил форму → получил один
`.docx` (`GenerateRequest = { templateCode, data }`, без хранилища).

Предложение возвращает Excel-логику оригинала, вынесенную за рамки MVP
(см. `ANALYSIS.md` §4, §8.4): «шапка объекта», таблица `База актов` (партия актов),
`Реестр ИД`, `Приложения_материалы.xlsx`. То есть переход к **stateful,
объект-центричной, пакетной** модели с персистентностью.

## Задачи по пунктам предложения

### Пункт 1 — Выбор объекта первым действием + автоподстановка
Сейчас первым идёт выбор типа документа (`TemplateSelector`). Нужна сущность «Объект»
с сохранённой шапкой проекта; при выборе объекта поля актов подтягиваются автоматически.

- [ ] 1.1. Модель данных `Object` (шапка проекта + справочники приказов/распоряжений и людей)
- [ ] 1.2. Хранилище объектов (см. «Решения»)
- [ ] 1.3. CRUD-эндпоинты: `GET/POST/PUT /api/objects`, `GET /api/objects/:id`
- [ ] 1.4. UI «Шаг 1: выбор объекта» перед выбором типа; экран создания/редактирования
- [ ] 1.5. Слой автоподстановки: поля объекта → закладки актов (наименование_объекта,
      реквизиты_*, подписанты)

### Пункт 2 — Подстановка приказов/распоряжений по «дате закрытия актов»
Новая логика (нет ни в коде, ни явно в оригинале): у приказов есть периоды действия,
для акта подставляется приказ, действовавший на дату акта.

- [ ] 2.1. В модель `Object` добавить приказы `{ номер, дата, действует_с, действует_по, тип }`
- [ ] 2.2. Функция выбора: по дате акта выбрать актуальный приказ/распоряжение
- [ ] 2.3. Подстановка результата в соответствующие закладки
- [ ] ⚠️ Уточнить семантику «дата закрытия актов» и привязку приказов к полям

### Пункт 3 — Реестр документов как контейнер для актов
Аналог таблицы `База актов` (одна строка = один акт).

- [ ] 3.1. Модель `Registry { id, objectId, items: ActEntry[] }`, `ActEntry = { templateCode, data }`
- [ ] 3.2. Хранилище реестров + эндпоинты `POST /api/registries`, `GET /api/registries/:id`, `PUT …`
- [ ] 3.3. UI: внутри объекта — список актов реестра, добавление/редактирование/удаление
      (текущий `DocumentForm` переиспользуется как редактор строки)
- [ ] 3.4. Присвоение и хранение ID реестра/актов

### Пункт 4 — Одна кнопка: сформировать весь реестр → готовые акты + готовый реестр
Пакетная генерация всех актов реестра + документа самого реестра, под ID.

- [ ] 4.1. Эндпоинт `POST /api/registries/:id/generate` — цикл по `items` (переиспользует `generateDocument`)
- [ ] 4.2. Генерация документа реестра (перечень всех актов). ⚠️ Формат: docx / xlsx / таблица
      (в оригинале `Реестр ИД.xlsx`)
- [ ] 4.3. Упаковка в один ZIP (акты + реестр) + отдача под ID
- [ ] 4.4. Frontend: кнопка «Сформировать реестр», прогресс, скачивание ZIP

### Пункт 5 — Авто-реестр «документов, подтверждающих качество» при >5 приложениях
Если в АОСР п.3 или АООК п.4 больше 5 приложений → отдельный реестр, все приложения
уходят туда, в акте — ссылка. Аналог `modReestrID` / `Приложения_материалы.xlsx`.

- [ ] 5.1. Парсинг списка приложений, подсчёт элементов
- [ ] 5.2. При `count > 5`: генерация отдельного документа-реестра приложений; в закладке
      акта (`приложения`/`Приложения`/`Приложения_АООК`) — ссылка вместо списка
- [ ] 5.3. Шаблон документа «реестр документов, подтверждающих качество». ⚠️ Нужен шаблон/формат
- [ ] 5.4. Подключение к пакетной генерации (п.4)

### Пункт 6 — Нумерация приложений в АОСР (1. 2. 3. …)
Самый локальный пункт; поле `Приложения` сейчас простой `textarea`.

- [ ] 6.1. Frontend: ввод приложений как список строк (добавить/удалить/перетащить)
- [ ] 6.2. Backend: автонумерация `1. … 2. …` при заполнении закладки (перенос строк через `<w:br/>`)
- [ ] 6.3. Согласовать с п.5 (нумерация vs вынос в реестр при >5)

## Решения, которые нужны до старта (блокируют)

1. **Персистентность.** SQLite/Postgres (полноценно) / JSON-файлы на диске (просто, но на
   Vercel ФС read-only → нужна внешняя БД) / только в сессии браузера. Влияет на весь
   бэкенд и на деплой.
2. **Источник данных объектов и приказов** — ручной ввод раз на объект или импорт из Excel?
3. **Семантика «по дате закрытия актов»** (п.2) — какая дата и как приказы привязаны к полям.
4. **Формат документов-реестров** (п.4 и 5) — docx / xlsx / таблица; есть ли исходные шаблоны.

## Влияние на текущий код

- `backend/src/types.ts` — новые модели `Object`, `Registry`, `ActEntry`.
- Новый слой хранилища (`backend/src/storage/`) + миграция на БД.
- Новые роуты `objects`, `registries`; `documents/generate` остаётся как primitive.
- `generatorService` — переиспользуется в цикле; добавляется ZIP-упаковка и генерация реестров.
- Frontend — новый flow: `Object → Registry → Acts`, текущие `TemplateSelector`/`DocumentForm`
  становятся вложенными шагами.
- Деплой: при выборе БД — добавить внешнее хранилище (Vercel ФС только для чтения).
